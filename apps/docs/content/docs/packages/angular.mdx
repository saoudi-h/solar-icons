The `@solar-icons/angular` package provides Angular components for Solar Icons. It's designed for Angular 17+ with modern patterns including signals, standalone components, and optimal performance.

## Installation

```package-install
@solar-icons/angular
```

## Features

- **Modern Angular 17+** → Uses signals (`input()`), standalone components, and optimized control flow
- **Minimal Bundle** → Tree-shakeable, only imported icons are included
- **Accessibility** → Native `@if` control flow for efficient title rendering
- **TypeScript** → Full type safety with IntelliSense support

## Basic Usage

Import icon components and use them in your templates:

```tsx
import { Component } from '@angular/core';
import { HeartBold } from '@solar-icons/angular';

@Component({
  selector: 'app-root',
  standalone: true,
  imports: [HeartBold],
  template: `
    <svg solarHeartBold [size]="24" color="#ef4444" />
  `,
})
export class AppComponent {}
```

## Import Patterns

All icons use **explicit global names** that include the style suffix (e.g., `HeartBold`, `ArrowLeftLinear`). This maps directly to the attribute selector you use in templates.

**From the root (any icon)**
```tsx
import { HeartBold, HeartLinear, HeartOutline } from '@solar-icons/angular';
import { StarBold, StarLinear } from '@solar-icons/angular';
```

**By category**
```tsx
import { HeartBold, HeartLinear } from '@solar-icons/angular/like';
import { ArrowLeftBold, ArrowRightLinear } from '@solar-icons/angular/arrows';
```

> **Note:** Per-style imports (`@solar-icons/angular/Bold`) are intentionally not provided. Angular icons are identified by their full global name in templates — importing `Heart` from `/Bold` would mislead you into writing `<svg solarHeart />` when the correct selector is `<svg solarHeartBold />`. Explicit names eliminate this ambiguity.

Each icon uses a camelCase attribute selector with the "solar" prefix:

```html
<svg solarHeartBold />
<svg solarAltArrowDownBold />
```

## Props

All icon components accept the following inputs:

| Input | Type | Default | Description |
|-------|------|---------|-------------|
| `size` | `string \| number` | `'1em'` | Icon size (e.g., `24`, `'2rem'`) |
| `color` | `string` | `'currentColor'` | Icon color (any valid CSS color) |
| `alt` | `string \| null` | `null` | Accessible label (adds `<title>` element) |
| `mirrored` | `boolean` | `false` | Flip icon horizontally |

### Examples

```tsx
// Size as number (pixels)
<svg solarHeartBold [size]="32" />

// Size as string
<svg solarHeartBold [size]="'2rem'" />

// Custom color
<svg solarHeartBold [color]="'#ef4444'" />

// Accessibility
<svg solarHeartBold [alt]="'Favorite'" />

// Mirrored
<svg solarArrowLeftBold [mirrored]="true" />

// Combined
<svg solarHeartBold 
  [size]="24" 
  [color]="'#ef4444'" 
  [alt]="'Add to favorites'" />
```

## Dynamic Rendering
When your icon names come from a database or need to be swapped at runtime, use the `SolarDynamicIcon` directive.

<Steps>
### Register your icons
To keep your bundle lean, you must explicitly register which icons should be available for dynamic lookup. You can do this at the application root or within a specific component.

```tsx
import { provideSolarIcons } from '@solar-icons/angular';
import { HeartBold, StarBold } from '@solar-icons/angular/like';

bootstrapApplication(App, {
  providers: [
    provideSolarIcons({ HeartBold, StarBold })
  ]
});
```

### Use the directive
Import `SolarDynamicIcon` and pass the icon name or the component class to the `solarIcon` input.

```tsx
import { SolarDynamicIcon, SunBold } from '@solar-icons/angular';

@Component({
  standalone: true,
  imports: [SolarDynamicIcon],
  template: `
    <!-- Lookup by registered name -->
    <ng-container [solarIcon]="'HeartBold'" />

    <!-- Direct class reference (no registration needed) -->
    <ng-container [solarIcon]="Sun" />
  `
})
export class App {
  protected readonly Sun = SunBold;
}
```
</Steps>

<Callout type="info" title="IDE Warnings">
When passing a component class directly, the Angular compiler might warn you that the component is unused (`NG8113`). This is a known limitation of static analysis with dynamic references; you can safely ignore it.
</Callout>

## Configuration & Props

Every icon component shares a consistent API for styling and accessibility:

| Input | Type | Default | Description |
|-------|------|---------|-------------|
| `size` | `string \| number` | `'1em'` | Numeric pixels or any CSS unit (e.g., `32`, `'2rem'`). |
| `color` | `string` | `'currentColor'` | Any valid CSS color value. |
| `alt` | `string` | `undefined` | Providing this adds a `<title>` and makes the icon accessible. |
| `mirrored` | `boolean` | `false` | Flip the icon horizontally. |

**Accessibility note:** When the `alt` input is provided, the component automatically renders a `<title>` element inside the SVG and removes `aria-hidden="true"`. Without `alt`, icons are marked as decorative with `aria-hidden="true"`.

## Optimization Tips

<Callout type="warn" title="Handling Large Grids">
If you are building a dashboard or an icon picker with hundreds of icons (500+), we recommend using **Static Components** instead of the dynamic directive.

For massive dynamic grids, always use `ChangeDetectionStrategy.OnPush` and pre-resolve your icon data into a `computed()` signal to keep your application fluid.
</Callout>
